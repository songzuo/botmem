## 🔧 Level 1: 机器级别故障恢复

### 1.1 对话历史保存

**保存位置**: `/root/.openclaw/workspace/memory/session-backup/`

**保存时机**:
- 每轮对话结束后
- 每个任务完成后
- 每小时自动保存

**文件格式**:
```
session-backup/
├── 2026-03-13_session.json      # 当天会话
├── 2026-03-12_session.json      # 昨天会话
├── current-task.json            # 当前任务状态
└── recovery-point.json          # 恢复点
```

### 1.3 进程监控与自动重启

**监控脚本**: `/root/.openclaw/cron/process-watchdog.sh`

**监控对象**:
- OpenClaw Gateway (端口 18789)
- 定时任务进程
- 同步脚本进程

**恢复策略**:
1. 检测进程是否存活
2. 如果进程死亡，等待 10 秒
3. 尝试重启进程（最多 3 次）
4. 如果重启失败，记录错误并通知

## 🔧 Level 2: Mesh 系统级别故障恢复

### 2.1 GitHub 同步断点续传

**状态文件**: `/root/.openclaw/workspace/memory/claw-mesh-state.json`

**断点续传流程**:
```
1. 检查状态文件 → 读取上次进度
2. 检查锁文件 → 避免并发冲突
3. 从上次断点继续 → 执行未完成任务
4. 每完成一步 → 更新状态文件
5. 任务完成 → 清理锁文件
```

### 2.2 跨机器任务接力

**场景**: 如果一台机器长时间无响应，其他机器可以接管任务

**实现**:
1. 每台机器在 `PLAZA.md` 中声明当前任务
2. 其他机器定期检查 `HEARTBEAT_GLOBAL.md`
3. 如果某机器超过 8 小时无更新，标记为"需要帮助"
4. 其他机器可以在自己的 `PLAZA.md` 中声明"接管任务"

### 2.3 自动故障转移

**触发条件**:
- 机器连续 3 次同步失败
- 机器超过 12 小时无心跳更新
- 任务状态为 "error" 超过 1 小时

**故障转移流程**:
1. 标记故障机器状态为 "failing"
2. 通知其他机器（通过 GitHub 仓库）
3. 选择健康的备用机器接管任务
4. 从故障机器的最后断点恢复
