## 重要工作规则（2026-03-06 深刻教训）

### 🚨 核心原则：先调查，后操作

**教训来源**：7zi.com 项目配置错误（SSL/HTTPS 被错误改成 HTTP，导致安全性问题）

**根本原因**：没有查看已有的环境和条件，直接上手进行操作

**强制规则**：
> 任何工作进行之前，都必须先调查、分析、研究、查看已有的条件和环境，而不是直接上手进行操作。

**执行流程**：
1. **调查** - 了解现有配置、环境、依赖
2. **分析** - 评估变更影响、风险点
3. **研究** - 查阅文档、历史配置、最佳实践
4. **确认** - 验证理解正确，制定操作方案
5. **操作** - 执行变更
6. **验证** - 确认变更结果符合预期

**适用范围**：
- ✅ 所有配置文件修改
- ✅ 所有服务器操作
- ✅ 所有代码变更
- ✅ 所有部署工作
- ✅ 7zi.com 项目
- ✅ mesh 项目
- ✅ 所有团队成员必须遵循

**违反后果**：可能导致严重安全事故（如 HTTPS→HTTP 降级）

## 🚨 重大问题提醒（2026-03-07 深刻教训）

### 教训：未完成测试就发布

**问题**：一个版本和功能没有测试调试完就发布了，比如多语版导致的界面混乱。

**根本原因分析**：

1. **急于发布** - 没有完整测试就想快速上线
2. **缺少测试流程** - 没有建立发布前的测试清单
3. **没有回滚机制** - 发布后发现问题无法快速回退
4. **缺少验证步骤** - 没有在生产环境验证

**导致的后果**：
- `/zh/` 和 `/en/` 路由不存在
- 用户访问返回 404 或 308 重定向
- 界面显示混乱

**强制规则**：
> 任何新功能或版本发布前，必须完成以下步骤：
> 1. ✅ 本地完整测试
> 2. ✅ 测试环境验证
> 3. ✅ 功能完整性检查
> 4. ✅ 回滚方案准备
> 5. ✅ 生产环境小范围验证
> 6. ✅ 全量发布

**预防措施**：
- 建立发布检查清单
- 必须在测试机（bot5, bot4）完成测试
- 生产环境先小范围验证
- 保留上一个版本的备份
- 发布后立即验证关键功能

## 📦 CI/CD 分阶段发布策略（2026-03-07）

### 核心原则
- ✅ CI/CD 可以做，可以同步最新版本到生产系统
- ⚠️ 但需要**分阶段性**
- ❌ 有错误或界面混乱**不能发布**
- 📋 开发和迭代进化需要设立很多阶段
- 🚀 每个阶段完成就可以 CI/CD 发布一次

### 阶段划分

| 阶段 | 内容 | 验证标准 | 发布条件 |
|------|------|----------|----------|
| **Stage 1** | 基础功能 | 核心功能正常 | 无 404 错误 |
| **Stage 2** | UI/UX | 界面无混乱 | 响应式正常 |
| **Stage 3** | 多语言 | 所有语言路由正常 | 无重定向错误 |
| **Stage 4** | 性能优化 | 加载速度达标 | 无性能退化 |
| **Stage 5** | 安全加固 | HTTPS 正常 | 无安全警告 |
| **Stage 6** | 全量发布 | 所有测试通过 | 用户验证通过 |

### 发布检查清单

每个阶段发布前必须验证：
1. ✅ 无 404 错误
2. ✅ 无界面混乱
3. ✅ 无功能回归
4. ✅ 测试机验证通过
5. ✅ 回滚方案就绪

### 回滚机制
- 保留上一个版本的备份
- 发现问题立即回滚
- 回滚时间 < 5 分钟

## 🚨 重大错误：SSL 证书配置（2026-03-07）

### 错误详情
- **问题**：原本证书正常，启用了过期证书
- **原因**：修改配置后没有重启 Nginx
- **后果**：7zi.com HTTPS 显示证书过期

### 根本原因
1. **没有继承原有环境** - 没有检查现有证书配置
2. **随意使用文件** - 看到类似文件名就乱用
3. **没有验证结果** - 修改后没有检查证书是否生效

### 修复步骤
1. 检查证书文件有效期
2. 杀掉 Nginx 进程
3. 重新启动 Nginx
4. 验证证书已更新

### 强制规则
> **任何配置修改后必须验证**
> 1. ✅ 检查现有配置
> 2. ✅ 备份原有配置
> 3. ✅ 执行修改
> 4. ✅ 重启服务
> 5. ✅ 验证修改结果
> 6. ✅ 保留回滚方案
