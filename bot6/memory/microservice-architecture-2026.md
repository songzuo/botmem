## 二、核心服务拆分方案

### 2.1 服务拆分原则

按照**业务边界 + 单一职责 + 高内聚低耦合**原则进行拆分，每个服务控制在 2-4 周能完成开发上线。

### 2.2 核心服务列表

| 服务名称          | 职责说明                              | 团队归属  | 峰值预估 QPS |
|-----------------|-------------------------------------|---------|------------|
| **API Gateway**   | 统一入口、路由、限流、认证、熔断         | 基础平台组 | 100,000   |
| **用户服务**       | 注册、登录、会员管理、地址管理           | 用户中心组 | 20,000    |
| **商品服务**       | SPU/SKU管理、分类、属性、上下架          | 商品中心组 | 15,000    |
| **搜索服务**       | 商品搜索、过滤、排序、热词              | 搜索组    | 30,000    |
| **购物车服务**     | 购物车增删改查、库存预占                | 交易组    | 10,000    |
| **订单服务**       | 订单创建、支付、取消、已完成             | 交易组    | 8,000     |
| **支付服务**       | 支付渠道接入、支付回调、对账             | 支付组    | 5,000     |
| **库存服务**       | 库存扣减、库存同步、库存预警             | 交易组    | 8,000     |
| **物流服务**       | 物流轨迹、快递查询、电子面单             | 履约组    | 3,000     |
| **评价服务**       | 商品评价、晒图、回复、点赞               | 商品中心组 | 5,000     |
| **通知服务**       | 短信、邮件、站内信、Push                 | 基础平台组 | 10,000    |
| **营销服务**       | 优惠券、满减、秒杀、拼团                 | 运营组    | 20,000    |
| **推荐服务**       | 个性化推荐、相关推荐、排行榜              | 算法组    | 15,000    |

### 2.3 服务间通信模式

- **同步通信**: gRPC（内部服务间，性能高，Protobuf序列化）
- **异步通信**: 事件驱动，通过 Kafka 发布/订阅
- **HTTP/REST**: 仅用于外部合作方接入或特定场景

## 三、数据库选型（SQL/NoSQL/缓存）

### 3.1 SQL 数据库

| 库名          | 选型         | 说明                              | 数据量预估   |
|-------------|------------|----------------------------------|-----------|
| **用户库**    | MySQL 8.0  | 主从+读写分离，TiDB可选              | 1000万用户  |
| **商品库**    | MySQL 8.0  | 分库分表（按商品ID哈希），3主3从        | 亿级SKU    |
| **订单库**    | MySQL 8.0  | 分库分表（按用户ID或订单ID），按月归档   | 亿级订单    |
| **支付库**    | PostgreSQL| 金融级事务支持，审计字段完备           | 千万级交易  |
| **营销库**    | MySQL 8.0  | 优惠券、活动配置表                   | 百万级活动   |

> **分库分表方案**: 采用 ShardingSphere-Proxy 或 MyCat，中间件代理层统一管理，初期按业务 ID 取模，后期扩缩容通过一致性哈希实现。

### 3.2 NoSQL 数据库

| 库名          | 选型           | 说明                              | 用途        |
|-------------|--------------|----------------------------------|-----------|
| **商品详情缓存**| Redis Cluster | 大 Value 存储（JSON序列化商品详情）    | 商品热数据缓存  |
| **搜索索引**    | Elasticsearch 7.x | 商品全文检索、聚合分析              | 搜索服务底层  |
| **地理位置**    | MongoDB 5.0   | 附近门店、配送范围查询               | LBS服务     |
| **行为日志**    | ClickHouse     | 用户行为分析、BI报表                | 数据分析     |
| **配置中心**    | Nacos           | 服务配置、开关、feature toggle       | 动态配置     |

### 3.3 缓存架构

```
L1 (本地缓存):   Guava Cache / Caffeine  → 进程内，热点数据（如分类字典）
L2 (分布式缓存): Redis Cluster 3主3从      → 跨进程， session、token、热榜
L3 (页面缓存):   Nginx Proxy Cache         → HTTP层，商品详情页HTML
```

**缓存策略**:
- **旁路缓存 (Cache-Aside)**: 先读缓存，缓存未命中查数据库并回填
- **延迟双删**: 写操作先删除缓存，再更新数据库，短暂延迟后再删除
- **Redis Cluster**: 3主3从，每主自动 RDB+AOF 持久化，主从同 Zone 部署

## 四、消息队列方案

### 4.1 选型: Apache Kafka vs RabbitMQ 对比

| 指标           | Kafka                         | RabbitMQ                      |
|--------------|------------------------------|------------------------------|
| **吞吐量**     | 百万级/秒                      | 万级/秒                        |
| **延迟**       | 毫秒级                        | 毫秒级                        |
| **消费模式**    | 发布/订阅（消费组）               | 队列 + 交换机                  |
| **消息持久化**  | 顺序写盘，TB级存储               | 内存+磁盘混合                   |
| **适用场景**    | 日志采集、事件流、数据同步           | 任务队列、服务解耦               |
| **最终选型**    | ✅ **Kafka**                  | 业务解耦 + 事件驱动             |

### 4.2 Kafka 集群架构

```
Topic: order-created, payment-success, inventory-changed, user-action

生产端: 服务 → Producer SDK → acks=all, retries=3
存储端: Broker 3节点 × 6分区 = 18分区，ISR副本=2
消费端: Consumer Group → 消费组内负载均衡
```

### 4.3 消息队列使用场景

| 事件                | 生产者      | 消费者                          | 作用                    |
|-------------------|-----------|-------------------------------|------------------------|
| `order.created`    | 订单服务    | 库存服务、支付服务、物流服务、通知服务   | 订单创建触发下游多服务处理      |
| `payment.success` | 支付服务    | 订单服务（更新状态）、库存服务（确认扣减） | 支付成功触发履约            |
| `inventory.changed`| 库存服务    | 搜索服务（更新可售状态）              | 库存变动同步搜索索引          |
| `user.action`     | 各服务     | 数据分析服务（行为日志入仓）            | 用户行为采集                |

### 4.4 可靠性保障

- **Producer**: acks=all + 重试3次 + 幂等发送（enable.idempotence=true）
- **Broker**: min.insync.replicas=2，副本因子=3
- **Consumer**: 手动提交 offset，失败重试（最多3次，死信队列兜底）

## 五、容器化和部署策略

### 5.1 容器化

- **容器**: 所有服务 Docker 化，基础镜像 alpine 或 distroless，控制镜像体积 < 200MB
- **镜像仓库**: 私有 Harbor 仓库，镜像 Tag 使用 Git Commit SHA，保证可溯源
- **编排**: Kubernetes (K8s) 1.28+，使用 Helm3 管理 Charts

### 5.2 服务网格 (Service Mesh)

引入 **Istio** 或 **Kuma** 作为服务网格，实现：
- **流量管理**: 熔断、重试、超时、故障注入
- **安全**: mTLS 服务间加密
- **可观测**: 自动 Jaeger/Prometheus 集成

### 5.3 部署拓扑

```
┌────────────────────────────────────────────────────┐
│                  Kubernetes 集群                    │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐           │
│   │ NodePool │ │ NodePool │ │ NodePool │           │
│   │  网关/入口 │ │  核心业务  │ │  数据/存储 │           │
│   │  (3节点)  │ │  (N节点)  │ │  (3节点)  │           │
│   └──────────┘ └──────────┘ └──────────┘           │
└────────────────────────────────────────────────────┘
         │                  │                │
    ┌────┴────┐       ┌────┴────┐       ┌────┴────┐
    │Dev环境   │       │Test环境 │       │Pre环境  │   │ Prod环境
    │ 1副本    │       │ 1副本   │       │ 1副本   │   │ 3+副本
```

### 5.4 部署策略

| 策略              | 说明                         | 适用场景          |
|----------------|-----------------------------|---------------|
| **滚动更新 (RollingUpdate)** | 逐台替换，旧→新，自动回滚          | 常规版本发布        |
| **蓝绿部署 (Blue-Green)**   | 两套环境，切换流量                 | 重大版本切换        |
| **金丝雀发布 (Canary)**      | 10%流量→50%→100%，监控先行        | 高风险变更        |
| **A/B Testing**             | 不同版本并行，灰度验证              | 策略变更验证        |

### 5.5 弹性伸缩 (HPA)

```yaml
# 基于 QPS + CPU 的混合伸缩策略
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      targetAverageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      targetAverageValue: 1000
```

## 六、监控和可观测性方案

### 6.1 可观测性三大支柱

```
可观测性 = 指标(Metrics) + 日志(Logs) + 链路追踪(Traces)
```

### 6.2 指标监控 - Prometheus + Grafana

| 指标类型    | 采集源              | 展示面板              |
|-----------|-------------------|-------------------|
| **基础设施** | node_exporter     | 节点 CPU/内存/磁盘/网络   |
| **中间件**  | kafka_exporter    | Kafka Producer/Consumer Lag |
| **应用**    | micrometer/Spring | JVM、QPS、RT、错误率     |
| **业务**    | 自定义埋点         | 订单量、支付成功率、搜索热词   |

**告警规则示例**:
- CPU > 80% 持续 5min → PagerDuty + 短信
- Kafka Consumer Lag > 10万 → 钉钉通知
- 订单错误率 > 1% → 立即告警

### 6.3 日志系统 - ELK/EFK 栈

```
服务日志(JSON格式)
    │
    ▼ (Fluent Bit 边车采集)
Kafka (缓冲层)
    │
    ▼
Elasticsearch (存储+检索)
    │
    ▼
Kibana / Grafana Loki (展示)
```

- **日志规范**: JSON格式，包含 trace_id, span_id, service_name, level, message, timestamp
- **采集**: 每个 Pod 挂载 Fluent Bit Sidecar，异步写入 Kafka 缓冲
- **存储**: Elasticsearch 7.x，冷数据 Index Lifecycle Management (ILM)，保留30天

### 6.4 链路追踪 - Jaeger + OpenTelemetry

- **埋点**: OpenTelemetry SDK 全链路自动注入，gRPC/HTTP 拦截
- **传输**: OpenTelemetry Collector → Jaeger Backend
- **采样**: 头部采样 100%，尾部采样异常请求
- **UI**: Jaeger Dashboard 查看调用链火焰图

### 6.5 统一可观测平台

推荐使用 **Grafana Stack** 一站式方案：

| 组件         | 职责               |
|------------|------------------|
| **Grafana**   | 可视化面板（指标+日志+追踪）  |
| **Prometheus**| 时序指标采集+存储        |
| **Loki**      | 日志聚合检索           |
| **Tempo**     | 分布式追踪存储          |
| **Alertmanager**| 告警聚合+路由          |

## 七、高可用与容灾设计

### 7.1 跨机房容灾

- **双机房部署**: 主机房 + 备机房，同城两中心，网络延迟 < 2ms
- **DNS 切换**: 故障时通过 DNSPod/阿里云 DNS 自动切换
- **数据库**: MySQL DDL 主从半同步，跨机房强一致性

### 7.2 熔断降级策略

| 场景              | 策略                       |
|-----------------|--------------------------|
| 下游服务超时        | gRPC 超时控制（Client: 500ms）  |
| 下游服务不可用      | 熔断器打开，断路器模式（失败率>50%） |
| 核心链路压力        | 限流（令牌桶，每服务独立）         |
| 非核心服务故障      | 降级，返回缓存数据或友好提示        |

## 九、架构总结

本方案基于**流量分层、存储分库、服务拆分、消息驱动**四大核心理念，构建支持日活 100万、峰值 QPS 10万的**高可用、高性能、高弹性**电商平台架构。

- **水平扩展**: 所有无状态服务均支持 K8s HPA 自动伸缩
- **容错设计**: 熔断、降级、限流多重保障，核心链路 99.99% 可用
- **可观测**: 指标+日志+链路三位一体，问题定位时间 < 5min
- **安全**: mTLS 服务间加密，WAF + DDoS 防护，JWT 统一认证
