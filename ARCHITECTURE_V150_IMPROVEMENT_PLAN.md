## 🔍 当前架构问题分析

### 问题 1: lib/ 层目录结构不统一

**现状**:

```
src/lib/
├── agent/              # 智能体核心模块
├── agent-scheduler/    # 智能体调度模块
├── a2a/               # A2A 通信模块
├── performance/        # 基础性能监控
├── performance-monitoring/  # 高级性能监控
└── ...
```

**问题描述**:

- 相关智能体模块（agent, agent-scheduler, a2a）分散在不同位置
- 两个性能模块（performance, performance-monitoring）功能重叠
- 缺乏统一的模块导出策略

**影响**:

- 导入路径混乱，降低开发效率
- 新人理解成本高
- 模块边界不清晰

---

### 问题 2: 性能模块功能重叠

**现状对比**:

| 模块                      | 功能                                   | 行数   | 状态   |
| ------------------------- | -------------------------------------- | ------ | ------ |
| `performance/`            | Web Vitals, 自定义指标, 基础预算       | ~1,500 | 基础版 |
| `performance-monitoring/` | 异常检测, 根因分析, 智能告警, 高级预算 | ~3,000 | 升级版 |

**重叠功能**:

- `performance/budget.ts` - 基础预算管理
- `performance-monitoring/budget-control/` - 高级预算配置

**问题描述**:

- 两个模块功能重复，造成混淆
- 不清楚应该使用哪个模块的预算功能
- 增加维护成本

---

   - [ ] 实现 `src/lib/events/event-bus.ts`
   - [ ] 实现 `src/lib/events/event-store.ts`
   - [ ] 支持内存和 Redis 两种实现

3. **集成到业务模块** (4h)
   - [ ] 智能体事件发布
   - [ ] 性能事件发布
   - [ ] 审计日志事件

4. **文档和示例** (3h)
   - [ ] 编写事件系统文档
   - [ ] 创建事件使用示例
   - [ ] 定义事件规范

**依赖关系**:

- 前置任务: P0-1, P0-3, P1-2

**验收标准**:

- ✅ 事件总线接口统一
- ✅ 关键业务事件已定义
- ✅ 事件消费者机制完善
- ✅ 文档和示例完整

**代码示例**:

```typescript
// 事件定义
export interface AgentEvent {
  type: 'agent.created' | 'agent.updated' | 'agent.deleted'
  payload: any
  timestamp: number
}

// 事件总线
export class EventBus {
  publish<T>(event: T): Promise<void>
  subscribe<T>(type: string, handler: (event: T) => void): Unsubscribe
}

// 使用示例
eventBus.publish<AgentEvent>({
  type: 'agent.created',
  payload: { id: '123', name: 'Agent' },
  timestamp: Date.now(),
})
```
