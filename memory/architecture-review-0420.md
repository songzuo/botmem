## 当前架构

### 1. 代理架构 (AGENTS.md)

采用**主管+子代理**两层模型：

```
主人
 └── 主管 (AI) — 任务分解、会议主持、汇报
      └── 11个子代理 — 各司其职 (架构师/Executor/测试员/等)
```

**优势:**
- 分工明确，专业化强
- 有会议机制和投票决策

**问题:**
- 子代理仅在任务时启动，无持久化运行态
- 主管依赖主人的直接指令，子代理之间缺乏直接协作通道
- 未定义子代理间的通信协议 (A2A? 直接调用?)

### 2. 记忆系统

**目录结构:**
```
memory/
├── YYYY-MM-DD.md          # 每日日志 (按日期)
├── daily/                 # 每日汇总 (结构化, 似乎未充分使用)
├── meetings/              # 会议记录 (仅1条)
├── *.md                   # 各类专题报告 (散乱)
├── claw-mesh-state.json   # 状态数据
└── heartbeat-state.json   # 心跳状态
```

**问题:**
- 日志文件过多 (60+ 个 .md 文件堆积在根目录)
- daily/ 和 meetings/ 子目录几乎为空，利用率极低
- 大量临时报告未清理 (如 `2026-03-22.md` ~ `2026-04-19.md` 跨度近一个月)
- 无统一的**知识库索引** — 找特定信息需遍历全部文件
- MEMORY.md 缺失 — AGENTS.md 要求有长期记忆文件但不存在

### 3. 核心定义 (SOUL.md)

定义了人格、沟通原则和连续性机制。**但:**
- IDENTITY.md 完全空白，角色身份未确立
- USER.md 只描述了主人信息，缺少用户画像和使用偏好

### 4. 工具配置 (TOOLS.md)

相对完整，包含:
- SSH 服务器清单 (3台，含Windows)
- TTS 配置
- SSH 密钥

**问题:**
- 某些信息可能已过时 (服务器IP、凭证)
- 缺少**部署目标**和**项目结构**的说明

## 改进建议

### 1. [P0] 重建记忆系统结构

**现状:** 60+ md文件堆在根目录，无法维护。

**建议:**
```
memory/
├── daily/          # 每日日志 (从 daily/YYYY-MM-DD.md 改为在此目录)
├── meetings/       # 会议记录
├── kb/              # 知识库索引 (按主题分类)
│   ├── architecture.md
│   ├── servers.md
│   └── projects.md
├── archive/         # 归档 (>30天的日志移至此)
└── MEMORY.md        # 长期记忆 (核心)
```

**行动:** 创建 `kb/` 目录，编写索引；将旧日志归档。

### 2. [P0] 补充缺失核心文件

- **MEMORY.md** — 长期记忆库，当前完全缺失
- **IDENTITY.md** — 填充角色身份定义
- **USER.md** — 补充主人偏好、使用习惯

### 3. [P1] 优化子代理协作机制

**现状:** 子代理只能通过主管中转，无法直接协作。

**建议:**
- 引入 **A2A (Agent-to-Agent)** 协议定义
- 允许同级别子代理直接通信 (如 架构师 ↔ Executor)
- 定义标准任务格式和结果回传契约

### 4. [P1] 引入心跳任务调度

**现状:** 无自动化的后台任务调度。

**建议:**
- 完善 `heartbeat-state.json` 使用
- 设置定期健康检查 (服务器/网站监控)
- 建立**cron任务表** (`memory/crontab.md`)

### 5. [P2] 清理和文档化

- 清理过期的专题报告
- 为 TOOLS.md 添加项目架构图
- 完善 docs/ 目录 (DEPLOYMENT.md、CHANGELOG.md 已有)
